# 阶段式目标：模块化端到端流式仿生人头系统

## 总体目标

本项目分阶段将当前整段式仿生人头链路升级为模块化端到端流式系统。

当前链路是：

```text
用户 wav 音频
  -> ASR 语音识别
  -> LLM 生成文本回复 + 情绪
  -> TTS 生成回复语音
  -> Morpheus 生成 52 维 blendshape
  -> UE5 获取逐帧权重并驱动数字人
```

阶段式建设目标是：

```text
第一步：稳定现有离线链路
第二步：建立评测、日志和数据采集体系
第三步：新增伪流式交互链路
第四步：升级为真流式 ASR / LLM / TTS / FaceDriver
第五步：用模块化 pipeline 生成数据，蒸馏实时 FaceDriver
第六步：联合优化 TTS 与 FaceDriver
第七步：探索更完整的 speech-to-speech-to-face 端到端模型
```

最终系统应同时保留：

```text
POST /pipeline/audio
用于离线测试、回归测试、数据生成和 teacher 数据采集。

WS /pipeline/stream
用于实时语音交互、音频 chunk 输入、回复 chunk 输出、blendshape chunk 输出和 UE5 实时播放。
```

### 首版 Provider 建设原则

首版同时建设 Mock provider 与真实 provider，不允许只完成其中一条线。

```text
自动化测试默认使用 Mock provider。
本地开发优先使用 Mock provider。
部署验收必须使用真实 provider。
/pipeline/stream 先用全 Mock 跑通协议、状态机、取消、失败和超时。
再依次接入 Ollama、faster-whisper、Piper、Morpheus 和 UE5 验证。
```

Mock provider 必须覆盖 ASR、LLM、TTS、Audio2Face 和 UE5 formatter，并支持可配置延迟、失败、超时、固定输出、取消和迟到结果。

真实 provider 固定为：

```text
ASR: faster-whisper base / CPU / int8 / zh
LLM: Ollama / qwen2.5:3b
TTS: Piper / zh_CN-huayan-medium
Audio2Face: Morpheus / conda env lyyMor
UE5: morpheus_52_raw
```

Mock 与真实实现必须遵循相同 adapter 接口，通过 JSON 配置切换，保持外部 HTTP 和 WebSocket 协议不变。

## 阶段 3：伪流式 pipeline 原型

### 阶段目标

新增实时交互接口，但不要求所有模型都是真流式。先把系统从“整段等待”升级成“分句生成、分句播放”。

这一阶段是最重要的工程过渡阶段。

### 目标架构

```text
用户音频 chunk 输入
  -> 服务端缓存为短 wav
  -> ASR 分段识别
  -> LLM token streaming
  -> 按句子切分回复
  -> TTS 按句子生成 wav
  -> Morpheus 按句子生成 blendshape
  -> UE5 按句子接收并播放
```

### 新增接口

```text
WS /pipeline/stream
```

### 事件类型

```text
client.audio.chunk
server.asr.partial
server.asr.final
server.llm.token
server.llm.chunk
server.tts.ready
server.face.frames
server.ue5.frames
server.pipeline.done
server.pipeline.error
```

### 主要任务

```text
1. 保留 POST /pipeline/audio 不变。
2. 新增 WS /pipeline/stream。
3. 前端或客户端按 chunk 上传音频。
4. 服务端做 VAD 或短音频缓存。
5. ASR 先采用分段 wav 识别，不强求真流式。
6. LLM 改为 token streaming。
7. LLM 输出按标点、句子或最大等待时间切 chunk。
8. Piper 按句子合成 wav。
9. Morpheus 按句子处理 TTS wav。
10. UE5 按句子接收 frames 并播放。
11. 先用全 Mock 验证 session_id、turn_id、cancel、error、done 和迟到结果丢弃。
12. 再按 Ollama、faster-whisper、Piper、Morpheus 的顺序替换真实 provider。
```

### 交付物

```text
1. /pipeline/stream WebSocket 接口。
2. stream_orchestrator.py。
3. streaming event schema。
4. LLM streaming adapter。
5. 句子级 TTS 调度器。
6. 句子级 Morpheus 调度器。
7. UE5 frame chunk 输出格式。
8. 全 Mock WebSocket 自动化测试。
9. 真实 provider WebSocket 冒烟测试。
```

### 验收标准

```text
1. 用户说完后，不必等待完整长回复生成完才开始播放。
2. LLM 生成第一句后，TTS 可以立即合成第一句。
3. 第一段 TTS wav 生成后，Morpheus 可以立即生成对应 blendshape。
4. UE5 可以先播放第一句，后续句子继续追加。
5. 老接口 /pipeline/audio 不受影响。
6. 被取消或过期 turn 的结果不会继续推送，也不会覆盖 latest。
7. 全 Mock 测试可重复，真实 provider 部署验收可独立执行。
```

## 阶段 4：真流式 ASR、TTS 与 Face 输出

### 阶段目标

将伪流式系统升级为真正的 chunk 级低延迟系统，减少整句或整段等待。

### 目标架构

```text
用户边说
  -> ASR 边输出 partial text
  -> LLM 边输出 token / phrase
  -> TTS 边输出 audio chunk
  -> FaceDriver 边输出 blendshape chunk
  -> UE5 边缓存边播放
```

### 主要任务

```text
1. 将 ASR adapter 扩展为 streaming interface。
2. 引入或替换支持 partial result 的 ASR。
3. 将 TTS adapter 扩展为 streaming interface。
4. 引入或替换支持低延迟 audio chunk 输出的 TTS。
5. 将 Audio2Face adapter 扩展为 chunk-level drive_stream。
6. 为每个 audio chunk 和 face frame chunk 加 timestamp。
7. UE5 端实现 jitter buffer。
8. 支持用户打断数字人说话。
9. 支持 turn-taking 状态机。
```

### 新增状态机

```text
IDLE
LISTENING
THINKING
SPEAKING
INTERRUPTED
ERROR
```

### 新增 streaming adapter 形式

```text
ASR:
  push_audio(chunk)
  emit asr.partial / asr.final

LLM:
  chat_stream(text)
  emit llm.token / llm.chunk

TTS:
  synthesize_stream(text_chunk, emotion, intensity)
  emit tts.audio.chunk

FaceDriver:
  drive_stream(audio_chunk, emotion, intensity)
  emit face.frames

UE5:
  send_frame_chunk(frames)
```

### 交付物

```text
1. streaming ASR adapter。
2. streaming TTS adapter。
3. streaming FaceDriver adapter。
4. WebSocket 双向事件协议。
5. UE5 jitter buffer。
6. 打断机制。
7. turn-taking 控制逻辑。
```

### 验收标准

```text
1. ASR 可以输出 partial text。
2. LLM 可以边生成边向 TTS 发送可说片段。
3. TTS 可以输出首个 audio chunk，而不是等完整 wav。
4. FaceDriver 可以输出连续 frame chunk。
5. UE5 播放不明显卡顿。
6. 支持用户在数字人说话时打断。
```

## 阶段 6：Morpheus 蒸馏为实时 FaceDriver

### 阶段目标

用 Morpheus 作为 teacher，训练一个轻量级 student FaceDriver，实现低延迟 audio chunk 到 52 维 blendshape chunk 的生成。

这是降低表情生成延迟的关键阶段。

### Teacher 模型

```text
response.wav + emotion + intensity
  -> Morpheus
  -> 52 维 blendshape frames
```

### Student 模型

```text
audio chunk + emotion + intensity
  -> future blendshape frames
```

### 主要任务

```text
1. 从 teacher dataset 中切分 audio chunk 和对应 blendshape chunk。
2. 设计 student FaceDriver 模型。
3. 输入音频特征、emotion embedding、intensity。
4. 输出未来 200ms 到 500ms 的 52 维权重。
5. 训练并验证 student。
6. 与 Morpheus 输出做对比。
7. 将 student 接入 Audio2Face adapter。
8. 支持 morpheus 和 student 两种 provider 切换。
```

### 推荐 provider 配置

```json
{
  "audio2face": {
    "provider": "student-facedriver"
  }
}
```

### 交付物

```text
1. student FaceDriver 训练代码。
2. 训练数据切分脚本。
3. 模型 checkpoint。
4. adapters/a2f_student_facedriver.py。
5. student 与 Morpheus 对比报告。
6. UE5 实时播放 demo。
```

### 验收标准

```text
1. Student 可以输出 52 维 blendshape。
2. 输出帧率稳定为 30fps 或可配置 fps。
3. 首帧延迟低于原 Morpheus 完整处理。
4. 嘴型同步质量接近 Morpheus。
5. 表情曲线平滑，无明显抖动。
6. 可以通过 provider 在 Morpheus 和 student 之间切换。
```

## 阶段 7：TTS 与 FaceDriver 联合优化

### 阶段目标

解决语音、韵律、情绪和面部动作之间的一致性问题，让数字人的声音和表情更加统一。

### 核心问题

当前模块化系统中，LLM 输出 emotion 和 intensity，TTS 与 FaceDriver 分别消费这些信息。但如果二者没有共享时间轴和韵律状态，可能出现：

```text
声音开心，但脸部表情不明显。
TTS 断句和面部动作断句不一致。
嘴型慢半拍或快半拍。
情绪强度变化不连续。
```

### 主要任务

```text
1. 建立 emotion state store。
2. 建立 prosody / timing representation。
3. TTS 输出 audio chunk 时同步输出 duration、phoneme 或 word timing。
4. FaceDriver 使用同一 emotion/prosody state。
5. 统一 TTS audio chunk 与 face frame chunk 的 timestamp。
6. 加入表情平滑和过渡机制。
7. 建立情绪一致性评测。
```

### 目标链路

```text
reply text chunk + emotion state
  -> TTS audio chunk + timing
  -> FaceDriver blendshape chunk
  -> UE5 synchronized playback
```

### 交付物

```text
1. emotion_state_store.py。
2. timing_alignment_service.py。
3. TTS timing metadata。
4. FaceDriver timing conditioning。
5. 情绪一致性测试集。
6. TTS-Face 同步评测报告。
```

### 验收标准

```text
1. 同一句回复中，语音情绪和面部表情一致。
2. 情绪强度变化能反映到面部动作幅度上。
3. 嘴型与语音时间基本同步。
4. 表情过渡平滑。
5. 不同 emotion / intensity 有可感知差异。
```

## 阶段 9：系统集成与最终交付

### 阶段目标

将离线测试链路、实时交互链路、teacher 数据生成、student FaceDriver 和 UE5 驱动整合成完整系统。

### 最终系统能力

```text
1. 支持完整离线链路：
   POST /pipeline/audio

2. 支持实时交互链路：
   WS /pipeline/stream

3. 支持模块替换：
   ASR / LLM / TTS / Audio2Face / UE5 formatter 均可通过 provider 切换。

4. 支持 teacher 数据生成：
   可批量生成 audio、text、emotion、tts、blendshape、ue5_json 数据。

5. 支持实时 FaceDriver：
   可由 audio chunk 生成 52 维 blendshape chunk。

6. 支持 UE5 实时播放：
   UE5 能按 timestamp 和 fps 播放 frame chunk。

7. 支持评测：
   能量化延迟、同步、口型质量、表情平滑度和情绪一致性。
```

### 最终交付物

```text
1. 完整服务端代码。
2. 完整 adapter 框架。
3. /pipeline/audio 离线接口。
4. /pipeline/stream 实时接口。
```

### 最终验收标准

```text

4. 语音与嘴型基本同步。
5. 面部表情与情绪一致。
6. UE5 播放稳定。
7. 模块可以独立替换。

```

## 一句话版本

本项目的阶段式目标是：

```text
先稳定现有 wav -> ASR -> LLM -> TTS -> Morpheus -> UE5 JSON 离线链路；
再建立日志、评测和 UE5 映射；
随后新增 /pipeline/stream，实现模块化流式端到端交互；
再用现有模块化 pipeline 批量生成 teacher 数据，蒸馏实时 FaceDriver；
最终实现低延迟、可替换、可评测、可 UE5 实时驱动的仿生人头语音-表情交互系统。
```
