## 0. 总体结论

本项目的阶段目标已根据低延迟与全双工调研更新。

新的主线不是立刻追求“电话级真全双工”，也不是马上硬改所有 provider 为真 streaming，而是把现有已跑通链路改造成：

```text
常驻模型
+ 可观测延迟指标
+ LLM 句子级增量
+ TTS / Face 句子或小块追赶
+ turn 级可取消
+ stale output 丢弃
+ 客户端回声消除
```

短期最关键的体验目标有两个：

```text
1. 数字人开始说话更快。
2. 用户一插话，数字人马上闭嘴，并且旧 turn 不再漏音频、漏脸部帧或覆盖 latest。
```

因此，当前阶段优先级调整为：

```text
P0: 稳定基线、mock/real provider、灰模预览、自动化测试。
P1: latency metrics、Ollama 预热常驻、Piper 常驻化。
P2: VAD / barge-in / playback.stop / generation_epoch。
P3: LLM streaming -> 句子级 TTS -> 句子级 Audio2Face。
P4: chunked ASR / partial transcript。
P5: UE5 实时接入、AEC、adaptive interruption。
P6: teacher 数据、FaceDriver 蒸馏、TTS-Face 联合优化。
```

## 5. 阶段 2：常驻模型与固定延迟消除

### 目标

不改变主协议，不急着做真流式 ASR，先消除最明显的固定延迟。

### 设计原则

```text
所有重模型尽量常驻。
所有 provider 支持 diagnostics。
所有 provider 输出前检查 turn_id / generation_epoch。
```

### Ollama 优化

```text
1. 服务启动或 session.start 时预热模型。
2. 使用 keep_alive 保持模型常驻。
3. 配置 num_ctx、num_predict、temperature。
4. chat_stream 继续使用流式输出。
5. 不再强制等完整 JSON 后才给 TTS。
```

推荐初始配置：

```json
{
  "llm": {
    "keep_alive": "30m",
    "num_ctx": 2048,
    "num_predict": 96,
    "temperature": 0.3
  }
}
```

### Piper 优化

当前 Piper CLI 每句起子进程是明显延迟来源。下一步应优先改为：

```text
方案 A: Piper Python API 常驻加载 voice。
方案 B: Piper HTTP server / sidecar 常驻。
方案 C: 后续 A/B CosyVoice2 / CosyVoice3 / LightTTS。
```

短期优先做 Piper 常驻化，不立即换 TTS 模型。

### 验收标准

```text
1. Ollama diagnostics 能体现模型已 ready。
2. LLM 首 token 延迟可观测。
3. Piper 不再每句重复加载模型。
4. tts_first_audio_ms 明显低于 CLI 子进程版本。
5. /pipeline/audio 和 /pipeline/stream 协议不破坏。
```

## 6. 阶段 3：VAD 与 barge-in 可打断

### 目标

先实现“用户一插话，数字人马上停”，这是全双工体验中比真 partial ASR 更重要的第一步。

### 状态机

```text
idle
listening
thinking
speaking
interrupt_pending
cancelled
recovering
```

### 核心语义

```text
用户 speech_start:
  -> server.turn.interrupting
  -> server.playback.stop
  -> cancel 旧 turn tasks
  -> generation_epoch += 1
  -> 旧 turn 后续结果全部 stale_drop
  -> 开始新 turn 或回到 listening
```

### 新增事件

```text
client.audio.vad.start
client.audio.vad.stop
client.playback.state
client.turn.interrupt

server.turn.started
server.turn.interrupting
server.playback.stop
server.turn.cancelled
server.turn.stale_drop
server.segment.accepted
server.segment.skipped
```

其中：

```text
server.turn.interrupting:
  控制语义，表示服务端进入打断处理。

server.playback.stop:
  播放语义，要求客户端 / UE5 立即停止旧音频和旧脸部播放。
```

### generation_epoch

每个 session 维护递增 epoch。

所有出站事件必须带：

```text
session_id
turn_id
generation_epoch
event_id
timestamp
```

出站前必须检查：

```text
如果 turn_id 或 generation_epoch 已过期:
  不发送给客户端
  不写 latest
  记录 server.turn.stale_drop
```

### VAD 策略

短期可用 RMS 能量阈值做最小实现，随后接入 Silero VAD。

推荐配置：

```json
{
  "vad": {
    "engine": "silero",
    "speech_start_ms": 120,
    "speech_end_ms": 450,
    "min_silence_ms": 350,
    "interrupt_min_speech_ms": 80
  }
}
```

### 客户端要求

如果客户端一边播放 TTS，一边继续开麦，则必须优先在客户端做回声消除：

```text
browser getUserMedia:
  echoCancellation: true
  noiseSuppression: true
  autoGainControl: true
```

服务端不应承担主要 AEC 责任。服务端只做 VAD、turn 控制、stale drop。

### 验收标准

```text
1. bot 说话时，用户插话会触发 playback.stop。
2. interrupt_to_playback_stop_ms 目标 < 250ms。
3. 旧 turn 不再发送 server.tts.audio。
4. 旧 turn 不再发送 server.ue5.frames。
5. 旧 turn 不再覆盖 latest。
6. 快速连续打断不会造成任务泄漏。
```

## 7. 阶段 4：句子级增量输出

### 目标

把当前伪流式进一步改造成真正有体验收益的“边生成边说”，但仍不要求 ASR 和 EmoTalk 真流式。

### 目标链路

```text
ASR final
  -> LLM streaming token
  -> sentence segmenter
  -> 第一句可说片段
  -> TTS 生成第一段音频
  -> 立即发送 server.tts.audio
  -> Audio2Face 按该音频生成 face frames
  -> 发送 server.ue5.frames
  -> 后续句子继续追加
```

### 句子切分策略

```text
按 。！？!?；; 切分。
segment_min_chars: 8~14。
segment_idle_flush_ms: 200~300。
sentence_max_chars: 60~100。
```

### LLM 输出策略

短期不建议让 LLM 先生成完整 JSON 再给 TTS。推荐：

```text
主流:
  LLM streaming 只负责自然语言 reply。

旁路:
  emotion / intensity 可用规则、轻量模型或尾部 metadata 更新。
```

### TTS 输出策略

短期：

```text
Piper 常驻 provider 按句子生成 wav。
server.tts.audio 仍可发送完整 wav chunk。
```

中期：

```text
Piper Python API 或替代 TTS 输出小 audio chunk。
```

### Face 输出策略

EmoTalk / Morpheus 当前按完整 wav 推理，不承诺真 streaming。

推荐做法：

```text
以 TTS segment 为 face segment。
每段优先 0.8~1.2s。
允许 face 相对 audio 滞后 100~300ms。
后续加 overlap 200~300ms + crossfade 平滑边界。
```

### UE5 frame chunk 协议

在 `bionic-head-ue5-v1 / morpheus_52_raw` 上增加时序字段：

```json
{
  "type": "server.ue5.frames",
  "session_id": "...",
  "turn_id": "...",
  "segment_id": "...",
  "generation_epoch": 12,
  "format": "morpheus_52_raw",
  "fps": 30,
  "start_ms": 1840,
  "frame_count": 27,
  "overlap_left": 6,
  "overlap_right": 3,
  "weights": [[0.0]]
}
```

### 验收标准

```text
1. LLM 第一可说句出现后立即进入 TTS。
2. 系统不等完整长回复结束才开始说话。
3. 第一段音频比旧版本更早到达客户端。
4. face 可以滞后音频 100~300ms，但不能长时间空脸。
5. 被打断后，后续句子全部取消或 stale drop。
```


### 验收标准

```text

2. partial 不频繁闪烁或大幅回滚。
3. final transcript 与旧完整 ASR 质量接近。
4. 初期 partial 只展示，不直接驱动 LLM。
```

## 12. 阶段 9：TTS-Face 联合优化与长期端到端

### 目标

进一步解决声音、韵律、口型、表情的一致性问题。

### 主要方向

```text
1. emotion state store。
2. prosody / timing representation。
3. TTS 输出音频时同步输出 duration / phoneme / word timing。
4. FaceDriver 使用同一 emotion/prosody state。
5. TTS audio chunk 与 face frame chunk 共享 timestamp。
6. 研究 speech-to-speech-to-face 端到端模型。
```

长期端到端模型只作为研究方向，不作为近期生产替代。近期生产主线仍是模块化 provider 架构。
